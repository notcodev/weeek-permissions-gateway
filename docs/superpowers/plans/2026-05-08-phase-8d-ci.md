# Phase 8d — CI Pipeline Hardening

**Goal:** Bring the GitHub Actions workflow from "runs lint+test" to a real release-gate that catches build regressions, image build regressions, and weekly dep drift.

**Architecture:** Single workflow `ci.yml`, three jobs (`static`, `test`, `docker`). Jobs run in parallel; PR checks pass only when all three are green.

---

### Task 0: Rewrite `.github/workflows/ci.yml`

**Files:** `.github/workflows/ci.yml`

**Changes vs. the previous version:**

1. **Build env placeholders** (`static` job).
   - `pnpm build` was a latent failure: `db/client.ts` and `aesGcm.ts` throw at module load when their env vars are missing, and Next 16 evaluates route modules during `next build` to collect page data. Same fix as the Dockerfile in 8c — supply syntactically-valid placeholders. postgres-js connects lazily, so the bogus URL is never opened.
   - This change is what made `pnpm build` actually pass in CI; before, it would have failed for any contributor without `.env.local` symbols already set in the runner.

2. **`docker` job** — verifies the Dockerfile builds on every PR with BuildKit cache via GHA cache backend (`type=gha`). No push, no registry login. Catches Dockerfile regressions before they ship.

3. **`concurrency`** group keyed on `github.ref` with `cancel-in-progress: true`. Force-pushes and rebase-fests stop racking up minutes.

4. **`permissions: contents: read`** at the workflow level — no job needs more.

5. **`timeout-minutes`** per job (15 / 25 / 20) so a hung job can't tie up a runner for the default 6 hours.

6. **Weekly schedule** (`0 6 * * 1`, Mondays 06:00 UTC) — cron run catches drift in transitive deps and base images even when nobody is pushing.

## Self-review

- YAML validated locally (`python3 -c "import yaml; yaml.safe_load(...)"`). Actionlint not installed locally — relies on GitHub's own validation on first push.
- Workflow not exercised end-to-end here (would require a push to a branch on the remote). Verification deferred to first push.
- The `docker` job uses `cache-from/cache-to: type=gha` — first run is cold, subsequent runs reuse the BuildKit layer cache.
- No security scanning step (Trivy / npm audit / CodeQL) in this phase. Opinionated trade-off: spec §18 doesn't require it for v1, and CodeQL needs `security-events: write` permissions which contradicts the least-privilege baseline. Add as a separate workflow later if compliance asks for it.
- No deployment job. Single-host compose deploy is operator-driven (per `docs/deploy.md`). A future phase could add an environment-gated `release` job that builds + pushes to GHCR and SSHes a `docker compose pull && up -d` to a host.
